# 插件管理与 Lua 开发指南

本文档描述 F17 插件 / Hook 系统当前已落地的能力、脚本编写约定，以及后续接入网关执行链时应保持的接口边界。

## 1. 当前状态

目前已落地：
- 设置页「插件 / Hook 管理」Tab，可创建、编辑、删除插件。
- 插件元数据持久化到 SQLite `plugins` 表。
- RPC 已支持 `plugin/list`、`plugin/upsert`、`plugin/delete`。
- 运行时、Hook 点、脚本内容、超时配置已有基础校验。
- 审计日志可记录 `plugin/upsert`、`plugin/delete` 写操作。

当前仍未落地：
- Lua 脚本实际执行。
- `pre_route` / `post_route` / `post_response` 挂载到网关主链路。
- 沙箱隔离与超时中断。

因此，当前阶段的插件更接近“注册表 + 脚本草稿管理”，不是已启用的生产执行链。

## 2. 数据模型

插件记录保存在 `plugins` 表，关键字段如下：

| 字段 | 说明 |
| --- | --- |
| `id` | 插件主键，未传时由后端生成 `plugin_xxx` |
| `name` | 插件名称，必填 |
| `description` | 插件说明，可选 |
| `runtime` | 当前仅支持 `lua` |
| `hook_points_json` | Hook 点数组，当前支持 `pre_route` / `post_route` / `post_response` |
| `script_content` | Lua 脚本文本，必填 |
| `enabled` | 是否启用 |
| `timeout_ms` | 预留给后续执行器的超时配置，范围 `1..=60000` |
| `created_at` / `updated_at` | 时间戳 |

## 3. Hook 点约定

PRD 规划的三个 Hook 点如下：

| Hook 点 | 触发阶段 | 目标 |
| --- | --- | --- |
| `pre_route` | 路由前 | 读取请求、做准入判断、拒绝请求或改写下游请求 |
| `post_route` | 选定账号后、发往上游前 | 根据命中账号、模型、路由元数据调整上游请求 |
| `post_response` | 收到上游响应后 | 补充标记、记录审计、改写响应体或附加元数据 |

当前仅做声明与持久化，不会真正执行。

## 4. 建议脚本结构

前端模板和后续执行器设计都默认 Lua 脚本暴露一个 `handle(ctx)` 函数：

```lua
function handle(ctx)
  return { action = "continue" }
end
```

建议遵循以下约定：
- 输入参数固定为 `ctx`。
- 无需改写时返回 `{ action = "continue" }`。
- 需要拒绝时返回 `{ action = "reject", status = 4xx/5xx, body = {...} }`。
- 仅做记录或标记时，优先返回 `annotations` 字段，而不是直接修改核心结构。

这些约定目前主要用于模板和文档对齐；真实执行器落地时应继续保持兼容，避免已保存脚本失效。

## 5. 预期上下文结构

为避免后续网关接入时反复改脚本，建议将 `ctx` 约束为 JSON 风格对象，并保持字段可缺省：

### `pre_route`

```json
{
  "request": {
    "path": "/v1/chat/completions",
    "method": "POST",
    "model": "gpt-5.3-codex",
    "stream": false
  },
  "apiKey": {
    "id": "key_xxx",
    "name": "默认 Key"
  },
  "account": {
    "id": "acc_xxx",
    "status": "active",
    "plan": "free",
    "primary_remain_percent": 12.5
  }
}
```

### `post_route`

```json
{
  "request": { "model": "gpt-5.3-codex" },
  "route": {
    "strategy": "weighted",
    "selectedAccountId": "acc_xxx"
  },
  "account": {
    "id": "acc_xxx",
    "plan": "plus"
  }
}
```

### `post_response`

```json
{
  "request": { "model": "gpt-5.3-codex" },
  "response": {
    "status": 200,
    "headers": {
      "content-type": "application/json"
    }
  },
  "route": {
    "strategy": "weighted",
    "selectedAccountId": "acc_xxx"
  }
}
```

说明：
- 以上字段属于建议契约，不代表当前后端已经注入。
- 所有字段都应允许缺省，脚本需自行判空。
- 敏感字段如 token、cookie、refresh token 不应进入脚本上下文。

## 6. 前端内置模板

设置页当前提供两个模板，适合用来约束后续执行器返回格式：

### 额度守卫

用途：当账号剩余额度低于阈值时，提前拒绝请求。

```lua
function handle(ctx)
  local account = ctx and ctx.account or nil
  local remain = account and account.primary_remain_percent or nil
  if remain ~= nil and remain < 5 then
    return {
      action = "reject",
      status = 429,
      body = {
        error = {
          message = "account quota too low",
          type = "quota_limit"
        }
      }
    }
  end
  return { action = "continue" }
end
```

### 响应审计

用途：当上游返回 5xx 时补充审计标记。

```lua
function handle(ctx)
  local response = ctx and ctx.response or nil
  local status = response and response.status or 0
  if status >= 500 then
    return {
      action = "continue",
      annotations = {
        severity = "error",
        tag = "upstream_5xx"
      }
    }
  end
  return { action = "continue" }
end
```

## 7. 推荐开发流程

1. 先在设置页创建插件草稿，明确名称、Hook 点和超时。
2. 使用内置模板起步，不要从空白脚本直接写复杂逻辑。
3. 将脚本设计成“字段缺失时安全退出”，避免未来接入执行器后因上下文不完整导致链路失败。
4. 在脚本里只处理业务判断，不依赖文件系统、网络或系统命令。
5. 如需扩展 `ctx` 字段，先同步更新 `docs/API.md` 与本指南。

## 8. 后续收口建议

下一阶段接入运行时时，建议按以下顺序推进：
- 引入 `mlua` 并在 `plugin/upsert` 阶段增加语法级校验。
- 先接通 `pre_route`，支持 `continue` / `reject` 两种结果。
- 再接通 `post_response` 的只读标记能力。
- 最后补沙箱隔离、超时保护和 `post_route` 请求改写。
