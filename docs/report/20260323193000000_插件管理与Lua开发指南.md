# 插件管理与 Lua 开发指南

本文档描述 F17 插件 / Hook 系统当前已落地的能力、脚本编写约定，以及现有网关执行链的 Hook 契约。

## 1. 当前状态

目前已落地：
- 设置页「插件 / Hook 管理」Tab，可创建、编辑、删除插件。
- 插件元数据持久化到 SQLite `plugins` 表。
- RPC 已支持 `plugin/list`、`plugin/upsert`、`plugin/delete`。
- `plugin/upsert` 会校验 runtime / hook 点 / 超时范围，并在保存前验证 Lua 脚本可加载且存在 `handle(ctx)`。
- 已接入 `mlua` 运行时，启用插件会在 gateway 请求链路中执行。
- 已启用 Lua 沙箱：禁用 `io` / `os` / `package` / `debug` / `require` / `loadfile` / `dofile` / `collectgarbage`。
- 已启用超时中断，脚本执行超出 `timeout_ms` 会被终止并记录运行时错误，不阻断主请求链路。
- 审计日志可记录 `plugin/upsert`、`plugin/delete` 写操作。
- 无启用插件时，网关通过进程内缓存快速短路，不会继续扫描插件表。

当前限制：
- `pre_route` / `post_route` 当前只支持改写 JSON 请求体和 `model` 字段，不修改路径、请求头或方法。
- `post_response` 当前以记录 `annotations` 和 trace 为主，不改写响应体。
- 插件异常会被记录并跳过，不会让整条请求直接失败；只有脚本明确返回 `action = "reject"` 才会终止请求。

## 2. 数据模型

插件记录保存在 `plugins` 表，关键字段如下：

| 字段 | 说明 |
| --- | --- |
| `id` | 插件主键，未传时由后端生成 `plugin_xxx` |
| `name` | 插件名称，必填 |
| `description` | 插件说明，可选 |
| `runtime` | 当前仅支持 `lua` |
| `hook_points_json` | Hook 点数组，当前支持 `pre_route` / `post_route` / `post_response` |
| `script_content` | Lua 脚本文本，必填 |
| `enabled` | 是否启用 |
| `timeout_ms` | Lua 执行超时（毫秒），范围 `1..=60000`，默认 `100` |
| `created_at` / `updated_at` | 时间戳 |

## 3. Hook 点约定

PRD 规划的三个 Hook 点如下：

| Hook 点 | 触发阶段 | 目标 |
| --- | --- | --- |
| `pre_route` | 路由前 | 读取请求、做准入判断、拒绝请求或改写下游请求 |
| `post_route` | 选定账号后、发往上游前 | 根据命中账号、模型、路由元数据调整上游请求 |
| `post_response` | 收到上游响应后 | 补充标记、记录审计、改写响应体或附加元数据 |

当前执行能力如下：
- `pre_route`：可读取 `request`、`apiKey`，返回 `continue` 或 `reject`，并可改写请求体 / `model`。
- `post_route`：可读取 `request`、`apiKey`、`route`、`account`，在发往上游前继续改写请求体 / `model`，或拒绝。
- `post_response`：可读取 `request`、`apiKey`、`route`、`account`、`response(status + headers)`，用于记录注解与 trace。

## 4. 建议脚本结构

当前执行器要求 Lua 脚本暴露一个 `handle(ctx)` 函数：

```lua
function handle(ctx)
  return { action = "continue" }
end
```

建议遵循以下约定：
- 输入参数固定为 `ctx`。
- 无需改写时返回 `{ action = "continue" }`。
- 需要拒绝时返回 `{ action = "reject", status = 4xx/5xx, body = {...} }`。
- 仅做记录或标记时，优先返回 `annotations` 字段，而不是直接修改核心结构。

当前后端已按这套契约执行，新增脚本应保持兼容。

## 5. 预期上下文结构

当前 `ctx` 为 JSON 风格对象，并保持字段可缺省：

### `pre_route`

```json
{
  "request": {
    "path": "/v1/chat/completions",
    "method": "POST",
    "model": "gpt-5.3-codex",
    "stream": false
  },
  "apiKey": {
    "id": "key_xxx",
    "name": "默认 Key"
  }
}
```

### `post_route`

```json
{
  "request": { "model": "gpt-5.3-codex" },
  "route": {
    "strategy": "weighted",
    "selectedAccountId": "acc_xxx"
  },
  "account": {
    "id": "acc_xxx",
    "plan": "plus"
  }
}
```

### `post_response`

```json
{
  "request": { "model": "gpt-5.3-codex" },
  "response": {
    "status": 200,
    "headers": {
      "content-type": "application/json"
    }
  },
  "route": {
    "strategy": "weighted",
    "selectedAccountId": "acc_xxx"
  }
}
```

说明：
- 所有字段都应允许缺省，脚本需自行判空。
- 敏感字段如 token、cookie、refresh token 不会进入脚本上下文。
- `request.body` 为 JSON 解析结果；非 JSON 请求体会回落为 `null`。

## 6. 前端内置模板

设置页当前提供两个模板，适合用来约束后续执行器返回格式：

### 额度守卫

用途：当账号剩余额度低于阈值时，提前拒绝请求。

```lua
function handle(ctx)
  local account = ctx and ctx.account or nil
  local remain = account and account.primary_remain_percent or nil
  if remain ~= nil and remain < 5 then
    return {
      action = "reject",
      status = 429,
      body = {
        error = {
          message = "account quota too low",
          type = "quota_limit"
        }
      }
    }
  end
  return { action = "continue" }
end
```

### 响应审计

用途：当上游返回 5xx 时补充审计标记。

```lua
function handle(ctx)
  local response = ctx and ctx.response or nil
  local status = response and response.status or 0
  if status >= 500 then
    return {
      action = "continue",
      annotations = {
        severity = "error",
        tag = "upstream_5xx"
      }
    }
  end
  return { action = "continue" }
end
```

## 7. 推荐开发流程

1. 先在设置页创建插件，明确名称、Hook 点和超时。
2. 使用内置模板起步，不要从空白脚本直接写复杂逻辑。
3. 将脚本设计成“字段缺失时安全退出”，避免因上下文差异触发误拦截。
4. 在脚本里只处理业务判断，不依赖文件系统、网络或系统命令。
5. 优先返回 `request.body` / `request.model` 的最小改动，不要一次性重写整份请求。
6. 如需扩展 `ctx` 字段，先同步更新 `docs/API.md` 与本指南。

## 8. 后续收口建议

后续可继续增强：
- 补充请求头、路径级改写能力，并明确允许改写的白名单字段。
- 为 `post_response` 增加可选的响应体改写能力，并限制为非流式响应。
- 给插件执行链补充更完整的 gateway 集成回归测试。
