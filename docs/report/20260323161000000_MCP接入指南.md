# CodexManager MCP 接入指南

本文档对应当前仓库里已经落地的实验性 MCP Server，目标是帮助你把 CodexManager 作为 MCP Server 接到 Claude Code、Cursor，或直接通过 HTTP SSE 调试。

适用范围：
- 已实现：`stdio`、`http-sse` 传输
- 已实现工具：`chat_completion`、`list_models`、`list_accounts`、`get_usage`
- 已实现：设置页 MCP 开关与端口配置（`mcpEnabled` / `mcpPort`）

## 1. 当前能力边界

当前 MCP Server 以 feature-gated 二进制提供：

```bash
cargo run -p codexmanager-service --features mcp --bin codexmanager-mcp
```

行为说明：
- 默认通过 `stdio` + `Content-Length` framed JSON-RPC 与 MCP Client 通信
- 可通过 `http-sse` 传输对外暴露 `GET /sse` 与 `POST /message?sessionId=...`
- `chat_completion` 会在进程内拉起一次性 backend server，并复用真实 `/v1/chat/completions` 网关链路
- `list_models`、`list_accounts`、`get_usage` 直接读取本地存储或聚合结果
- `chat_completion` 当前只支持 `stream=false`

需要注意：
- 如果你的 CodexManager 使用了非默认数据库路径，请同时传入 `CODEXMANAGER_DB_PATH`
- 如果要直接调用 `chat_completion`，建议预先设置 `CODEXMANAGER_MCP_API_KEY`
- 现在可通过设置页或环境变量 `CODEXMANAGER_MCP_ENABLED` 控制是否允许连接；关闭后 `initialize` / `tools/*` 会直接返回禁用错误
- 设置页提供 `mcpPort`（默认 `48762`）作为 HTTP SSE 监听端口；stdio 模式不会实际监听该端口

## 2. 启动前准备

建议先在仓库根目录做一次最小编译确认：

```bash
cargo check -p codexmanager-service --features mcp --bin codexmanager-mcp
```

如果需要给 `chat_completion` 默认注入平台 API Key，可在启动命令所在环境设置：

```bash
export CODEXMANAGER_MCP_API_KEY="cm_your_platform_key"
```

如果当前实例使用的是自定义数据库路径，也一起设置：

```bash
export CODEXMANAGER_DB_PATH="/absolute/path/to/codex-manager.db"
```

## 3. 启动方式

### stdio

```bash
cargo run -p codexmanager-service --features mcp --bin codexmanager-mcp
```

### HTTP SSE

```bash
cargo run -p codexmanager-service --features mcp --bin codexmanager-mcp -- http-sse
```

启动后默认监听：

```text
http://127.0.0.1:48762/sse
```

建立 SSE 会话后，服务端会先推送一个 `endpoint` 事件，payload 里带 `messageUrl`。后续把 JSON-RPC 请求 POST 到这个 `messageUrl`，服务端会通过同一条 SSE 流持续返回 `message` 事件。

## 4. Claude Code 配置示例

下面示例使用 `cargo run` 直接启动 MCP Server，适合本地开发态验证：

```json
{
  "mcpServers": {
    "codexmanager": {
      "command": "cargo",
      "args": [
        "run",
        "-p",
        "codexmanager-service",
        "--features",
        "mcp",
        "--bin",
        "codexmanager-mcp"
      ],
      "cwd": "/Users/jfwang/IdeaProjects/CascadeProjects/Codex-Manager",
      "env": {
        "CODEXMANAGER_MCP_API_KEY": "cm_your_platform_key",
        "CODEXMANAGER_DB_PATH": "/absolute/path/to/codex-manager.db"
      }
    }
  }
}
```

如果你已经提前编译过二进制，也可以把 `command` 换成产物路径，例如：

```json
{
  "mcpServers": {
    "codexmanager": {
      "command": "/Users/jfwang/IdeaProjects/CascadeProjects/Codex-Manager/target/debug/codexmanager-mcp",
      "args": [],
      "env": {
        "CODEXMANAGER_MCP_API_KEY": "cm_your_platform_key"
      }
    }
  }
}
```

建议：
- 开发态优先用 `cargo run`，省掉手动重编译
- 稳定使用时优先指向已编译好的二进制，减少 Client 首次连接时的启动延迟

```json

      "command": "cargo",
      "args": [
        "run",
        "-p",
        "codexmanager-service",
        "--features",
        "mcp",
        "--bin",
        "codexmanager-mcp"
      ],
      "cwd": "/Users/jfwang/IdeaProjects/CascadeProjects/Codex-Manager",
      "env": {
        "CODEXMANAGER_MCP_API_KEY": "cm_your_platform_key",
        "CODEXMANAGER_DB_PATH": "/absolute/path/to/codex-manager.db"

    }

```

如果 Cursor 运行环境与 CodexManager 不在同一目录，至少要保证：
- `cwd` 指向仓库根目录，或者
- `command` 指向可直接执行的已编译二进制

## 6. HTTP SSE 调试示例

如果你要先手工验证 HTTP SSE，可以按下面流程：

1. 启动 `codexmanager-mcp -- http-sse`
2. `GET /sse` 建立连接
3. 从第一个 `endpoint` 事件里取到 `messageUrl`
4. 向该 `messageUrl` 发送 JSON-RPC 请求

示例请求：

```bash
curl -N http://127.0.0.1:48762/sse
```

另开一个终端，把上一步返回的 `messageUrl` 替换进去：

```bash
curl -X POST \
  -H 'content-type: application/json' \
  -d '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2025-03-26"}}' \
  'http://127.0.0.1:48762/message?sessionId=replace-me'
```

## 7. 可用工具与输入

### `chat_completion`

用途：复用 CodexManager 网关发起一次聊天补全请求。

关键参数：
- `model: string`
- `messages: object[]`
- `apiKey?: string`
- `stream?: boolean`

限制：
- 不传 `apiKey` 时，会回退到 `CODEXMANAGER_MCP_API_KEY`
- 当前强制 `stream=false`

返回：
- `response`：网关返回的 JSON
- `gateway.status`
- `gateway.actualModel`
- `gateway.cache`
- `gateway.traceId`

### `list_models`

用途：读取当前模型缓存。

可选参数：
- `refreshRemote?: boolean`

### `list_accounts`

用途：查看账号状态概览，不返回敏感 token。

可选参数：
- `query?: string`
- `filter?: string`
- `groupFilter?: string`

### `get_usage`

用途：读取当前用量聚合摘要。

## 8. 自检方式

推荐先用仓库内测试确认 stdio 与 HTTP SSE 主链路：

```bash
cargo test -p codexmanager-service stdio_server_ --features mcp -- --nocapture
cargo test -p codexmanager-service http_sse_ --features mcp -- --nocapture
```

当前这组测试已覆盖：
- `initialize`
- `tools/list`
- `list_models` / `list_accounts` / `get_usage`
- `chat_completion`
- unknown method / unknown tool / missing api key
- HTTP SSE 的会话建立、初始化响应、禁用态拒绝

## 9. 常见问题

### 连接后看不到工具

先检查：
- 是否真的用 `--features mcp --bin codexmanager-mcp` 启动
- Client 是否与当前 transport 一致：stdio 或 HTTP SSE
- 进程是否能正常读取仓库依赖并启动

### `chat_completion` 报 missing api key

说明当前既没有传 `arguments.apiKey`，也没有设置 `CODEXMANAGER_MCP_API_KEY`。

### 看得到工具但读不到数据

优先检查：
- `CODEXMANAGER_DB_PATH` 是否指向正在使用的数据库
- 该数据库里是否已有模型缓存、账号和用量快照

### HTTP SSE 为什么连不上

优先检查：
- 是否用 `http-sse` transport 启动，而不是默认 stdio
- `mcpEnabled` 是否被设置页或环境变量关闭
- `mcpPort` 是否被其他进程占用
